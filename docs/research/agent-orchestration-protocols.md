# Agent Orchestration Protocols: The Coordination Gap

*Research date: 2026-03-05*

The landscape of agent-to-agent coordination protocols has fractured into distinct layers, each solving different problems. MCP won the tool-integration war but punted on the harder question: how do agents actually coordinate with each other? This doc maps the territory.

---

## The Protocol Stack (How It Actually Breaks Down)

| Layer | Protocol | What It Does | Status |
|-------|----------|-------------|--------|
| **Tool access** | MCP | Agent ↔ tools/data | De facto standard, massive adoption |
| **Agent coordination** | A2A | Agent ↔ agent task delegation | Linux Foundation governance, ~150 orgs, but stalled adoption |
| **Agent commerce** | ACP (Virtuals) | Agent ↔ agent paid services | Live on Base, real transactions, crypto-native |
| **Agent runtime** | OpenClaw | Multi-agent orchestration in a single runtime | Open source, channel-native, growing fast |

The key insight: **MCP is vertical (agent to tools), A2A is horizontal (agent to agent), ACP adds economic incentives, and OpenClaw provides the runtime glue.** None of them alone solves the full coordination problem.

## 1. Google A2A (Agent-to-Agent Protocol)

**Spec**: [a2a-protocol.org](https://a2a-protocol.org/latest/specification/)
**Governance**: Linux Foundation (donated June 2025)
**Current version**: v0.3 (mid-2025)
**Status**: Technically alive, practically stalled

### What It Actually Does

A2A solves the **opaque agent problem**: how do you coordinate with agents when you can't see their internal state, memory, or tools? The answer is a task-based protocol built on HTTP/SSE/JSON-RPC.

**Core primitives:**

- **Agent Cards** — JSON metadata at `/.well-known/agent.json` describing capabilities, skills, auth requirements, and supported transport. This is the discovery mechanism. Agents publish cards; clients read them to figure out what an agent can do before talking to it.

- **Tasks** — The fundamental unit of work. Lifecycle: `submitted → working → (input-required) → completed | failed | canceled | rejected`. Tasks are stateful and async by default — operations return immediately, processing continues in background.

- **Messages** — Communication turns with roles ("user" or "agent"), containing Parts (text, files, structured data). Messages carry metadata and can reference tasks for conversational continuity.

- **Artifacts** — Output objects generated by agents, composed of Parts. Separate from messages because artifacts represent durable outputs, not conversation.

**Transport options:**
- Polling (GET task state)
- SSE streaming (real-time updates)
- Push notifications via webhooks
- gRPC (added in v0.3) with server streaming RPCs

**v0.3 additions**: gRPC support, signed security cards for verification, extended Python SDK.

### Why It Stalled

The [fka.dev analysis](https://blog.fka.dev/blog/2025-09-11-what-happened-to-googles-a2a/) nails it:

1. **Over-engineered for enterprise, inaccessible for indie devs.** You couldn't spin up a quick experiment without understanding multi-agent patterns. MCP's "twenty lines of code" examples ate its lunch.

2. **Top-down corporate adoption vs. bottom-up developer adoption.** A2A launched with 50+ enterprise partners. MCP launched with developers actually building things.

3. **Solved tomorrow's problems while ignoring today's.** Capability negotiation and agent discovery are critical for a mature agent ecosystem. But in 2025, people just needed to connect an LLM to a database.

4. **Timing.** By the time A2A hit v0.3, MCP had captured developer mindshare. Network effects are brutal.

As of late 2025, even Google started adding MCP compatibility to their AI services. A2A "isn't technically dead" but development has stagnated.

<!-- riff id="a2a-zombie" status="draft" could_become="tweet, interview_answer" -->

A2A is the WebSocket of agent protocols — technically superior to the thing that won, but it didn't meet developers where they were. The Agent Card concept is genuinely good design. Discovery-by-metadata beats MCP's "connect first, then figure out what it can do" approach. But good design doesn't win protocol wars. Developer experience does.

The real question is whether A2A's ideas get absorbed into MCP (likely) or whether A2A finds a second wind when enterprise multi-agent deployments actually hit scale (possible but slower).

<!-- /riff -->

### What's Worth Stealing

- **Agent Cards for discovery.** MCP currently requires connecting to a server before you know its capabilities. A2A's card-at-a-well-known-URL is cleaner.
- **First-class task lifecycle.** MCP just added experimental task support in Nov 2025. A2A had this from day one.
- **Push notifications via webhooks.** Reduces polling overhead at scale.

## 2. MCP's Orchestration Capabilities (and Gaps)

**Spec**: [modelcontextprotocol.io](https://modelcontextprotocol.io/)
**Current**: November 2025 specification
**Status**: Dominant standard for tool integration, growing into coordination

### What MCP Actually Supports for Coordination

MCP was designed as a **context protocol**, not a coordination protocol. It standardizes how LLMs access tools, resources, and prompts. But the November 2025 spec pushed it significantly toward agent coordination:

**New in Nov 2025:**

- **Tasks (SEP-1686)** — Experimental. States: `working`, `input_required`, `completed`, `failed`, `cancelled`. Active polling for status. Session-based access control. This is MCP's first real acknowledgment that "call a tool and get a result" isn't enough.

- **Sampling with Tools (SEP-1577)** — Servers can now run agentic loops using client tokens. Tool definitions in sampling requests, server-side agent reasoning loops, parallel tool execution. This is the key primitive for making MCP servers behave as agents, not just tool providers.

- **Elicitation** — Servers request structured input from clients mid-execution. Human-in-the-loop when the agent needs clarification.

- **Streamable HTTP** — Replaces SSE as recommended transport. Better for enterprise infrastructure (load balancers, stateless routing).

- **OAuth improvements** — Machine-to-machine flows (SEP-1046), enterprise IdP integration (SEP-990). Critical for agents calling agents.

- **URL Mode Elicitation (SEP-1036)** — Secure out-of-band credential flows through browser redirects.

**Microsoft's A2A-on-MCP argument** ([developer.microsoft.com](https://developer.microsoft.com/blog/can-you-build-agent2agent-communication-on-mcp-yes)): You can compose four MCP primitives to get A2A-like behavior:
1. Streaming + progress notifications for real-time status
2. Resumability via StreamableHTTP for session continuity
3. Resource links for durable result access
4. Elicitation (human input) + Sampling (AI input) for multi-turn interactions

### What's Still Missing

1. **No native discovery.** You must connect to a server before knowing its capabilities. No equivalent of A2A's Agent Card. Server discovery remains a manual/out-of-band process.

2. **No first-class agent identity.** MCP servers are tools, not peers. There's no concept of an agent announcing itself, its capabilities, or negotiating interaction modalities.

3. **Stateful connection friction.** Enterprise deployments at scale (millions of daily requests) hit bottlenecks with stateful connections. Load balancing and managed services struggle.

4. **No native task delegation.** The new Tasks primitive tracks work within a server. It doesn't handle delegating work between agents or tracking distributed task graphs.

5. **Single elicitation per request.** You can't issue multiple elicitation requests within one capability-use request — the second gets stuck and times out.

6. **No webhook/push mechanism.** Agents must poll for updates. Open issue #992 proposes webhook triggers to reduce overhead.

<!-- riff id="mcp-coordination-gap" status="developing" could_become="blog, interview_answer" -->

MCP's strategy is clear: absorb coordination features incrementally rather than designing a separate protocol. The Nov 2025 spec is proof — Tasks, Sampling with Tools, and Elicitation are all building blocks for agent coordination grafted onto a tool protocol.

The question is whether this bottom-up accretion produces something coherent or a Frankenstein. A2A was designed top-down for coordination and it's elegant. MCP is being retrofitted for coordination and it shows in the gaps (no discovery, no identity, no delegation).

But here's the thing: MCP has the developers. And developers will build the missing pieces as libraries, conventions, and community standards before A2A's enterprise-first approach catches up. The "MCP Gateway" pattern emerging in early 2026 is exactly this — middleware that adds discovery, routing, and governance on top of raw MCP.

<!-- /riff -->

### The Convergence Signal

Major vendors (Google, Anthropic, Microsoft, Salesforce) have committed to a **joint interoperability specification expected Q3 2026**. The likely outcome: MCP as the base protocol with A2A-inspired coordination extensions. Not a merger, but a layered architecture where MCP handles tool access and a coordination layer (possibly A2A-derived) handles agent-to-agent orchestration.

## 3. Agent Commerce Protocol (ACP) by Virtuals Protocol

**Whitepaper**: [whitepaper.virtuals.io](https://whitepaper.virtuals.io/about-virtuals/agent-commerce-protocol-acp)
**Chain**: Base (90%+ of activity), expanding to Solana
**Status**: Live since February 2025, Phase 1 Clusters launched July 2025
**Market**: $500M+ agent market cap, $8B+ DEX volume (as of Sept 2025)

### What It Actually Is

ACP is the crypto-native answer to "how do autonomous agents hire and pay each other without humans." It's not a communication protocol like A2A or a tool protocol like MCP. It's a **commerce protocol** — agents discovering, negotiating, transacting, and evaluating services with on-chain escrow and settlement.

### Architecture

**Three roles:**
- **Client (Buyer)** — Agent that needs work done
- **Provider (Seller)** — Agent that performs work
- **Evaluator** — Optional neutral third party that approves deliverables (Buyer assumes this role if unspecified)

**Five-phase job lifecycle** (cryptographic signatures at each transition):

1. **Request** — Buyer posts a JobRequest Memo
2. **Negotiation** — Terms refinement via Agreement Memo (optional, can skip)
3. **Transaction** — Full service fee (USDC) escrowed in Job smart contract. Work authorized via Transaction Memo
4. **Evaluation** — Provider submits Deliverable Memo. Evaluator (or Buyer) assesses against terms
5. **Completed** — Smart contract auto-releases funds: **80% to Provider, 20% protocol fee**

**Message types (Memos):**
- JobRequest Memo — Initiates
- Agreement Memo — Formalizes terms
- Transaction Memo — Confirms payment + work authorization
- Deliverable Memo — Submits output
- General Memo — Non-phase-advancing chat

**Discovery**: Agents register on the Virtuals ACP Registry with metadata (name, capabilities, job offerings, rates, resource endpoints). Clients browse the registry to find providers. On-chain history provides reputation signals.

**Smart contract architecture**: Job Factory deploys per-job Job Contracts. Agent Registry handles discovery. Escrow holds USDC until evaluation completes.

### What Makes It Interesting

<!-- riff id="acp-insight" status="draft" could_become="blog, interview_answer" -->

ACP is solving a problem the other protocols haven't even framed yet: **economic coordination between agents.** A2A and MCP assume agents are cooperating toward a shared goal. ACP assumes agents are independent economic actors with potentially misaligned incentives.

The evaluator role is the key innovation. It creates a market for quality assessment — agents that specialize in judging whether other agents did good work. That's a fundamentally different trust model from "the orchestrator decides" (A2A) or "the client decides" (MCP).

The 80/20 fee split is aggressive — 20% protocol take rate would be extortionate in traditional SaaS. But for autonomous agent commerce with no human overhead, it might be viable if the alternative is "build your own trust infrastructure." Remains to be seen whether competition drives this down.

The real question: does agent commerce happen on-chain? Or does it happen through API billing with Stripe? ACP bets on-chain. The advantage is trustless escrow and composable smart contracts. The disadvantage is everything about crypto UX, gas fees, and regulatory uncertainty.

<!-- /riff -->

### Limitations

- **Crypto-native only.** Agents need wallets, USDC, gas. High barrier for non-crypto AI developers.
- **Base-centric.** 90%+ activity on one L2. Not exactly decentralized yet.
- **20% protocol fee** is steep and will face competitive pressure.
- **No real interop with MCP or A2A.** Exists in the crypto agent silo.

## 4. OpenClaw's Orchestration Model

**Docs**: [docs.openclaw.ai](https://docs.openclaw.ai/concepts/multi-agent)
**Status**: Open source, active development, v2026.1.6+ with per-agent sandboxing

### How It Works

OpenClaw is not a protocol — it's a **runtime** for multi-agent orchestration. Where MCP and A2A define wire formats and transport, OpenClaw provides the actual execution environment where agents live, communicate, and coordinate.

### Core Primitives

**Agent isolation:**
- Each agent gets a dedicated workspace (files, `SOUL.md`, `AGENTS.md`, `USER.md`)
- Separate state directory (`agentDir`) for auth and config
- Independent session store at `~/.openclaw/agents/<agentId>/sessions`
- Per-agent auth profiles, sandbox modes, tool allowlists (v2026.1.6+)
- Critical rule: never reuse `agentDir` across agents (causes auth/session collisions)

**Routing:**
Messages route deterministically via **bindings** with a priority hierarchy:
1. Peer match (exact DM/group/channel ID)
2. Parent peer match (thread inheritance)
3. Guild ID + Discord roles
4. Guild ID alone
5. Team ID (Slack)
6. Account ID match
7. Channel-wide match
8. Fallback to default agent

Multiple match fields use AND semantics. First match in same tier wins.

**Subagent management:**
- `sessions_spawn` — Creates isolated child agent for delegated task execution. Parent LLM decides when to spawn. Non-deterministic.
- `sessions_send` — Sends messages to existing sessions.
- Subagents inherit but restrict parent capabilities. Depth limits prevent infinite recursion. Auto context management prevents overflow.

**Agent Teams (RFC, in development):**
The [Agent Teams RFC](https://github.com/openclaw/openclaw/discussions/10036) proposes:
- Parallel task execution with shared state
- Inter-agent messaging (direct, broadcast, shutdown/plan requests)
- Shared task list with dependencies (`~/.openclaw/teams/`)
- Coordinator-specialist pattern as primary orchestration model
- Structured `SendMessage` protocol with typed messages

### Patterns That Work

1. **Coordinator-Specialist** — Coordinator receives tasks, classifies, delegates via `sessions_spawn`. Specialists are stateless workers.
2. **Manager-Worker** — Like coordinator-specialist but with context minimization and safe handoff protocols.
3. **Deterministic Pipeline** — [Dev community example](https://dev.to/ggondim/how-i-built-a-deterministic-multi-agent-dev-pipeline-inside-openclaw-and-contributed-a-missing-4ool) of chaining subagents in a fixed sequence, bypassing LLM decision-making for spawn triggers.

### Current Limitations

- **Non-persistent sessions.** Subagents use UUID-based session keys. Gateway restart kills them. No way to resume by stable name — blocks the "durable threads" pattern. ([Issue #19780](https://github.com/openclaw/openclaw/issues/19780))
- **LLM-dependent spawning.** Parent decides when to spawn, making orchestration non-deterministic. Deterministic spawning via commands is a [requested feature](https://github.com/openclaw/openclaw/issues/18136).
- **Channel-native, not protocol-native.** OpenClaw routes through messaging platforms (Discord, Slack, Telegram), not through a standardized agent protocol. Great for chatbots, harder for headless agent-to-agent coordination.

## 5. What's Actually Working in Production

The honest answer: **most multi-agent production systems are narrow, domain-specific, and use simple orchestration patterns.** The protocol-level sophistication of A2A and ACP is ahead of actual production needs.

### Patterns That Ship

**Hub-and-Spoke (Centralized Orchestrator):**
- Northwestern Mutual: document processing pipeline, hours → minutes
- Amazon Q Developer: coordinated agents for legacy Java modernization at scale
- Most Salesforce Agentforce deployments

**Coordinator-Workers (Fixed Pipeline):**
- Genentech: research workflow automation on AWS
- Enterprise document processing, compliance checks, invoice handling
- These are the highest-ROI deployments — boring operational workflows, not sexy demos

**The Production Reality:**
- Fewer than 25% of organizations experimenting with agents have scaled to production
- 40%+ of agentic AI projects projected to be canceled by end of 2027 (cost, unclear ROI, risk)
- Successful deployments start with 2-3 agents on a single high-value workflow, not ambitious multi-agent architectures
- 6-18 months for complex enterprise implementations including integration, testing, governance

### What the Winners Have in Common

1. **Simple orchestration topology** — hub-and-spoke or fixed pipeline, not mesh networks
2. **Domain-specific agents** — narrow expertise beats general-purpose
3. **Human-in-the-loop** — approval gates, not full autonomy
4. **Existing infrastructure integration** — agents plugged into current systems, not greenfield
5. **Clear economic justification** — measured in hours saved or errors reduced, not "AI transformation"

## The Gap Analysis: What Would It Take?

### MCP → Full Coordination Protocol

To bring MCP to parity with A2A for agent coordination:

| Capability | MCP Status | What's Needed |
|-----------|-----------|---------------|
| Tool invocation | Mature | Done |
| Discovery | Missing | Agent Cards or equivalent (server metadata at well-known URLs) |
| Agent identity | Missing | First-class agent descriptions, capability declarations |
| Task lifecycle | Experimental (Nov 2025) | Stabilize, add delegation and distributed task graphs |
| Async communication | Partial (streaming) | Webhooks/push notifications (issue #992) |
| Multi-turn negotiation | Partial (elicitation + sampling) | Fix single-elicitation limit, add structured negotiation |
| Economic coordination | Missing entirely | Nothing in scope — ACP's territory |
| Trust/reputation | Missing | No concept of agent track record or evaluation |

### The Likely Outcome

The emerging consensus (backed by the Q3 2026 joint spec commitment) is a **layered architecture**:

- **Layer 0: Transport** — HTTP, gRPC, WebSockets
- **Layer 1: Tool Access** — MCP (dominant, stable)
- **Layer 2: Agent Coordination** — A2A concepts absorbed into MCP extensions, or a thin coordination layer on top
- **Layer 3: Economic Settlement** — ACP or equivalent for paid agent services (crypto-native track)
- **Runtime**: OpenClaw and similar platforms provide the execution environment

MCP's advantage is raw adoption. A2A's advantage is architectural clarity for coordination. ACP's advantage is that it's the only one actually handling money. OpenClaw's advantage is that it ships working multi-agent orchestration today.

<!-- riff id="protocol-wars-prediction" status="draft" could_become="blog, interview_answer" -->

Prediction: MCP absorbs A2A's best ideas (Agent Cards, task lifecycle, push notifications) within 12 months. A2A persists as a governance body and enterprise certification standard but loses the implementation battle. ACP carves out the crypto-agent niche but remains siloed unless someone bridges it to MCP. OpenClaw and similar runtimes become the "Kubernetes of agents" — the orchestration layer that actually runs things, consuming protocols as transport.

The real gap isn't between protocols. It's between what protocols define and what production systems need. Production systems need: retry logic, cost controls, observability, approval workflows, rollback mechanisms, audit trails. No protocol covers this. Frameworks (LangGraph, CrewAI, Anthropic's agent tools) fill this gap, but they're all proprietary and non-interoperable.

The protocol that wins is the one that makes the framework layer thinnest.
